# 01-基础概念：从 Docker 到 Kubernetes

## 1. 为什么我们需要 Kubernetes？
我们在单体时代或者微服务初期，使用 `Docker` + `Docker Compose` 足够应对单台机器的部署。但是当服务节点增加，我们需要解决以下问题：
- **故障自愈**：如果某个 Docker 容器挂了，如何自动拉起新的？一台物理机宕机了，它上面的容器如何漂移到健康的机器？
- **弹性伸缩**：如何根据 CPU/内存使用率自动增加前台服务的实例数？
- **负载均衡与服务发现**：微服务实例动态增减，内部应用之间如何互相发现和调用？（Spring Cloud 虽有 Eureka/Nacos，但 K8s 在基础设施层解决，不仅限于 Java）。
- **配置与密钥管理**：如果不重新构建镜像，如何更新 application.yml？

**Kubernetes 就是解决这些容器集群调度和管理问题的操作系统。**

## 2. 宏观架构 (Master-Node 模型)
K8s 集群分为两部分：**控制面 (Control Plane / Master)** 和 **工作节点 (Worker Node)**。

### 2.1 控制面 (是大脑)
- **API Server**：所有指令的入口，只认 RESTful API (或 kubectl)。
- **ETCD**：分布式键值数据库，保存集群的所有状态数据（"大脑的记忆"）。
- **Scheduler**：调度器，决定一个新建的 Pod 应该扔到哪台 Node 上运行。
- **Controller Manager**：控制器管理器，负责维护集群的状态，比如确保一直有指定数量的 Pod 在运行。

### 2.2 工作节点 (是手脚)
- **Kubelet**：Node 的节点代理，向 Master 汇报状态，听从 Master 命令创建/销毁容器。
- **Kube-proxy**：负责节点上的网络路由规则（网络代理），实现 K8s Service 的负载均衡。
- **Container Runtime**：底层容器运行时（如 Docker, containerd）。

## 3. 核心概念对比 (面向 Java 开发者)
| 概念 | 在传统 Java 架构中的类似物 | 在 K8s 中的作用 |
|------|-----------------------|---------------|
| **Pod** | 一台虚拟机/Tomcat实例 | K8s 调度的最小单元，里面可以包含一个或多个容器（通常是一个 Java 应用容器）。 |
| **Deployment** | 进程守护/扩容脚本 | 告诉 K8s 你的应用需要几个副本，负责应用的无停机滚动更新与回滚。 |
| **Service** | Nacos/Ribbon + 本地 Nginx | 为一组 Pod 提供一个固定的内部访问 IP (ClusterIP) 和负载均衡。 |
| **Ingress** | Nginx Gateway / Spring Cloud Gateway | 集群的外部请求入口，根据 HTTP 域名或路径将请求路由到不同 Service。 |
| **ConfigMap** | Spring Cloud Config / Nacos Config | 将非敏感配置文件外置，通过挂载注入到 Pod 中。 |

## 4. 为什么最小调度单元是 Pod 而不是容器？
Pod 的设计理念类似“豌豆荚”，里面包含多个紧密协作的容器。
**典型场景（Sidecar 模式）**：
一个 Pod 里运行两个容器：
1. **主容器**：你的 Spring Boot 服务 (8080端口)。
2. **Sidecar 容器**：Filebeat 日志收集器。
它们共享网络命名空间（可以通过 `localhost` 互访）和存储卷卷（共同读写日志目录）。这样实现了**业务逻辑（Java）与基础设施逻辑（日志收集）的解耦**。
