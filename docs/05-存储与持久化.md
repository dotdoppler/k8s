# 05-存储与持久化：把数据库部署在 K8s 上

在 Java 后端开发中，业务数据是无价的。前面的 `demo-service` 属于**无状态服务 (Stateless)**，它可以被随意起停销毁。而数据库 (如 MySQL, Redis) 属于**有状态服务 (Stateful)**，如果 Pod 重启了导致数据丢失，那就是巨大的生产事故！

在 K8s 中，容器所在的文件系统是临时的（Ephemeral）。这就需要引入 K8s 强大的存储卷挂载机制。

## 1. 核心存储概念
为了解耦“想要多少硬盘空间”和“底下是用云盘还是物理本地盘”，K8s 设计了三层模型：

1. **PV (PersistentVolume, 持久卷)**：由集群管理员（运维负责人）在各种底层存储（比如 AWS EBS、阿里云盘、NFS）上划出的一块真实网盘空间。
2. **PVC (PersistentVolumeClaim, 持久卷声明)**：作为业务开发者，你不知道也不必关心底层是什么盘，你只需要向集群声明："**我的 MySQL 需要一个 1GB 且可以读写配置的持久卷**"。
3. **StorageClass (存储类)**：运维不再手动帮你建 PV，而是建一个 StorageClass。当业务端申请 PVC 时，StorageClass 侦测到需求，自动动态给你创建一个一模一样大的 PV 并绑定（称为 Dynamic Provisioning，Minikube 默认带了一个叫 `standard` 的 StorageClass）。

## 2. MySQL 持久化实战
我们需要为 MySQL 部署以下资源：
1. **一个 PVC** (`mysql-pv-pvc.yaml`)：申请 1GB 空间。
2. **一个 Service**：在集群内暴露 3306 端口。
3. **一个 Deployment** (`mysql-deployment.yaml`)：配置 MySQL 环境变量，并将 PVC 作为卷挂载到容器内的 `/var/lib/mysql` 数据存放目录。

（注：在实际生产的复杂集群中，我们通常采用 `StatefulSet` + `Headless Service` 加 Operator 的模式来部署高可用主从数据库集群。但对于单节点数据库或作为学习对象，Deployment + PVC 已经能够完美展示存储原理）。
