# 02-实战部署：第一个 Spring Boot 到 K8s

在这节实战中，我们将一个典型的完整 Java 容器化部署流程在本地跑通。

## 1. 我们的目标应用 (apps/demo-service)
我们在 `apps/demo-service` 下准备了一个极简的 Spring Boot Web 应用。
**关键信息**：
- 它暴露了 `GET /hello` 接口，并且会打印当前运行的 Pod 主机名。
- 采用 **多阶段构建 (Multi-stage Build)** 的 `Dockerfile`，将 Maven 编译和 JRE 运行环境分离，这是企业级应用的常见做法（避免将庞大的 JDK 和源码打包进最终镜像）。

## 2. K8s 清单文件解析 (Manifests)
我们要将应用跑在 K8s 上，需要告诉集群我们想要什么。这种“描述期望状态”的文件就是 Manifests，用 YAML 书写。在 `manifests/basic/` 目录下准备了两个文件：

### A. Deployment (`demo-deployment.yaml`)
负责拉起并管理我们的服务实例：
- `replicas: 3`: 我们要求集群始终保持有 3 个应用的副本（Pod）在运行。如果其中一个挂了，K8s 会自动拉起一个新的。
- `imagePullPolicy: Never`: 因为我们是在本地用 Minikube 构建的镜像，告诉 K8s 不要去网上下载，直接用本地环境里的 `dev/demo-service:v1`。这能避免出现 ErrImagePull 错误。

### B. Service (`demo-service.yaml`)
负责为前面那 3 个 Pod 提供稳定的访问入口和负载均衡：
- 哪怕 Pod 挂了再拉起导致 IP 变了，Service 的内网 IP (ClusterIP) 永远不变。
- `type: NodePort` 允许我们从集群外部（比如你的浏览器）访问服务。我们指定了外网访问端口 `30080`，你可以通过这扇门进入集群内部。

## 3. 部署脚本剖析 (scripts/deploy-demo.sh)
由于我们在 Mac 上运行，Docker 守护进程通常在外部，而 Minikube 集群在它自己的虚拟机/容器里。
为了让 K8s 找到我们刚 build 的镜像，脚本中使用了：
```bash
eval $(minikube docker-env)
```
这行命令极大地简化了本地测试环节：“劫持”了本地终端的 Docker 配置，让它直接在 Minikube 环境内部进行镜像构建，免去了推送到外部 Registry 的麻烦。

接下来，执行 `kubectl apply -f xxx.yaml` 就相当于是按下了启动按钮。

```bash
./scripts/deploy-demo.sh
```
