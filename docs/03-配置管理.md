# 03-配置管理：ConfigMap 与代码解耦

在传统的架构中，我们常常使用 Spring Cloud Config 或者 Nacos 来作为集中式配置中心。云原生应用的一个核心理念是 **不可变基础设施 (Immutable Infrastructure)**：当你需要修改数据库密码或环境标志时，绝对不应该去重新构建 Docker 镜像，而是让镜像去挂载外部配置。

## 1. 概念理解
- **ConfigMap**：专门用于存储非敏感的键值对（例如 `application.yml`，日志级别，外部 API 地址等）。
- **Secret**：用于存储敏感的键值对（例如数据库账号密码、TLS 证书），内容在 ETCD 中会被保护，拉取时自动 Base64 解码。

## 2. Java / Spring Boot 如何读取 ConfigMap？
有两种最主流的方式：
1. **环境变量注入 (Environment Variables)**：这是最灵活直接的方式。在 K8s 中将 ConfigMap 注入环境，Spring Boot 有强大的松散绑定机制，会自动将环境变量 `APP_MESSAGE` 映射为 `@Value("${app.message}")`。
2. **挂载为文件 (Volume Mount)**：可以把 ConfigMap 中存储的整个 `application-prod.yml` 挂载到容器内的 `/config/` 目录下，Spring Boot 启动时会自动识别。

我们在接下来的实战中，采用 **方式一 (环境变量)** 来演示应用如何在不修改代码的情况下，实现多环境的不同输出！

## 3. 核心清单改造解析
我们新增了 `manifests/config/demo-configmap.yaml`：
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: demo-service-config
data:
  APP_MESSAGE: "🌟 Hello from K8s ConfigMap (V2)!"
```

并且修改了 Deployment (`demo-deployment-v2.yaml`)：
```yaml
      containers:
      - name: demo-service
        image: dev/demo-service:v2   # 我们打了一个新版本的镜像
        envFrom:                     # 魔法在这里：把配置字典里的所有键值对，都变成环境变量！
        - configMapRef:
            name: demo-service-config
```

## 4. 架构师进阶解析：K8s 原生配置 vs Nacos/Apollo 传统微服务配置中心

很多初学者会疑惑：在 Spring Cloud 体系中，我们可以通过 Nacos 结合 `@RefreshScope` 做到应用不重启的热更新。为什么我们演示的 `envFrom` (环境变量注入方式) 需要通过 Pod 重启来拉取新配置？是不是 K8s 比较落后？

这里涉及到底层的理念差异和两种组件在现代架构中**互补共生**的最佳实践：

### A. K8s ConfigMap (基础设施级：不变性)
- **更新机制**：环境变量固化于 Linux 进程启动之时，不可热发变。要想动态更新，必须将 ConfigMap 挂载为**文件卷 (Volume)** 进入容器，配合 `spring-cloud-starter-kubernetes-config` 进行文件系统事件监听才能实现热更新。
- **最佳场景**：存放**基础设施级别、不常变动**的“静态”配置（如数据库密码、外部微服务基础域名、甚至 Nacos 的地址），它符合 K8s 提倡的**不可变基础设施 (Immutable Infrastructure)**原则。配置一旦出错，可以连同整批 Pod 直接无脑 `kubectl rollout undo` 快速回滚。

### B. Nacos/Apollo (业务级：高机动性)
- **更新机制**：基于长连接的心跳和推送，毫秒级下发，框架底层自动更新 Bean。
- **治理优势**：K8s ConfigMap 只能算作 ETCD 中的一个原生文本存储，而 Nacos/Apollo 拥有极其完善的用户权限管理、版本历史、灰度发布 (针对部分机器下发新配置观察) 以及极好的 UI 大盘。
- **最佳场景**：存放**业务运行时的动态配置**（如营销活动的开关、限流熔断的调参、根据时段改变的大促策略）。

**总结结论**：
在企业级 K8s 架构中，这两者不是"谁干掉谁"的关系。架构师往往会选择**两者结合**：K8s ConfigMap 提供应用启动**最底层赖以生存的根环境要素**，而 Nacos/Apollo 承载应用启动后**复杂的动态业务调参**。

## 5. 实战篇：如何更新采用 envFrom 注入的 ConfigMap？
当我们在 `demo-configmap.yaml` 中修改了 `APP_MESSAGE` 的值后，需要分为两步让微服务生新配置：

1. **更新 K8s 数据库中的 ConfigMap**：
   ConfigMap 本质上只是存在 K8s 控制平面 (ETCD) 的一条记录，通过以下命令即可立即在集群侧更新：
   ```bash
   kubectl apply -f manifests/config/demo-configmap.yaml
   ```

2. **触发微服务的滚动重启 (Rollout Restart)**：
   由于采用环境变量注入 (`envFrom`)，Java 进程已经在启动瞬间读取并固化了旧的变量。为了让服务加载新变量，我们必须通过下述命令让 Deployment **安全且零停机地重新拉起 Pod**：
   ```bash
   kubectl rollout restart deployment demo-service-deployment
   ```
   *（注：这条命令会逐个销毁旧 Pod，并用含有新环境变量的容器拉起新 Pod，从而实现业务无缝衔接的热更新体验。）*
