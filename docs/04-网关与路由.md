# 04-网关与路由：Ingress 暴露外部流量

作为 Java 后端开发者，我们对网关并不陌生（比如 Nginx、Spring Cloud Gateway、Zuul）。而在 K8s 的体系中，承担**集群级统一 7 层流量路由**重任的组件，就叫做 **Ingress**。

## 1. 为什么需要 Ingress？
之前我们学习了 `Service (NodePort)`，它能把内部服务暴露出来，但它有明显的缺点：
1. **端口受限**：必须要占用 30000+ 的随机/指定端口，这对于真实的域名访问 (`http://api.myapp.com:80`) 是不可接受的。
2. **缺乏七层路由**：NodePort 是纯四层转发。如果在集群里又部署了个用户中心 `user-service`，你只能再开一个 NodePort 端口。如果你想实现按域名或按 URL Path (`/api/v1/demo`, `/api/v1/user`) 分发到不同 Service，NodePort 束手无策。

**Ingress 的出现就是为了解决这些问题**：
它像一个集群级别的 "大 Nginx"，它可以监听 80 和 443 端口，根据你在 `Ingress` 资源中定义的规则（域名、路径等），把 HTTP 流量智能切分到对应的各个 `Service` 上。

## 2. Ingress 的架构 (Controller 与 Resource)
很多初学者常在这里摔跟头：`Ingress` 本质上只是你在 K8s 里写的一份 **YAML 路由规则单**。集群本身是不懂怎么处理它的。
要想让这份路由单生效，集群里必须安装一个真实跑在后台的反向代理程序，这叫 **Ingress Controller**（最主流的实现是 Nginx Ingress Controller）。

## 3. 实操：为我们的 Demo 服务绑定域名
我们的目标是：
1. 在本地电脑的 `/etc/hosts` 中加入一行隐射：`127.0.0.1 demo.local.com` (Mac 的 Minikube 环境通过隧道透传给本地端口)。
2. 配置 Ingress，让发往 `demo.local.com` 的请求全部打到我们之前的 `demo-service` 上。

我们会创建一份名为 `demo-ingress.yaml` 的文件：
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: demo-ingress
spec:
  rules:
  - host: demo.local.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: demo-service
            port:
              number: 8080        # 注意这里填的是 Service 内部映射到后端容器的 TargetPort 或者 Service 自己的 Port
```
你会发现，这份配置和你手写 nginx.conf 的 `server_name` 和 `location` 非常相似！

## 4. 终极实战辟坑：当本地环境“脑裂”时如何排查？
在前面的实战演练中，很多同学（尤其是 Mac/Windows 开发者）会在本地遇到 Ingress 返回 **502 Bad Gateway** 的问题。这是 K8s 网关学习中最宝贵的一课：

**A. 网络劫持 (为什么不用 `minikube tunnel` 和本地 `/etc/hosts`)**
当你访问 `http://demo.local.com/hello` 报错时，往往不是 Ingress 挂了，而是你电脑上跑的其他软件（如 Clash 代理、Docker 自带的代理拦截）提前把 80 端口的请求劫持了，它们找不到路由，所以返回了 502。
这就是微服务开发中常说的**环境污染**。

**B. 降维打击破解法 (`kubectl port-forward`)**
为了彻底避开本地乱七八糟的网络代理，我们采用了强行建隧道的手段：
```bash
kubectl port-forward -n ingress-nginx service/ingress-nginx-controller 8080:80
```
**原理**：K8s API Server 利用长连接，强行在你本机开了一个 `8080` 端口，任何发进这个端口的流量，都会被加密打包，直接送到集群内部 `ingress-nginx` 这个真实 Pod 的 80 端口上！这就叫“**指哪打哪**”。

**C. 流量的全生命周期 (从浏览器到 Java 代码)**
1. **浏览器发起** -> `http://127.0.0.1:8080/hello`
2. **K8s 隧道传输** -> 被 `kubectl` 进程捕获，送进 Minikube 虚拟机内的 Ingress Pod (80端口)
3. **Nginx Ingress 匹配** -> 由于我们在 Ingress Yaml 里删掉了 `host: demo.local.com` 且 `path: /`，这个请求完美命中无差别通配规则。
4. **Service 负载均衡** -> Nginx 查到了你名叫 `demo-service`，发现它底下挂着 3 个 Java Pod，随机挑了一个 `10.244.0.23:8080`。
5. **Spring Boot 接收** -> 流量来到 Tomcat，匹配到 `@GetMapping("/hello")`，返回 HTTP 200 和带有 😈 的字符串返回给前端！
